# Basement Tablet Dashboard — Design

**Date:** 2026-07-19
**Status:** Approved by operator (design); implementation not started

## Purpose

A wall-mounted ~10" landscape tablet in the basement replaces the wall switches for five
lighting zones. It lives in the main living space, so it must be aesthetically pleasing,
and it must be operable by anyone with zero explanation — tap a tile, the lights toggle.

## Scope

In scope: one new Lovelace dashboard, one input_number helper, one light group helper,
two automations, build/deploy tooling in this repo.

Out of scope: the existing `basement-control` dashboard (untouched), wallpanel/kiosk
screensaver configuration, bathroom fan, scene buttons, any PAE integration.

## Dashboard

New **storage-mode dashboard** `basement-tablet`, title "Basement". Generated by a build
script (like the weather dashboard) — never hand-edited in HA.

### Layout (10" landscape, layout-card)

│  ┌──────────────┐  ┌──────────────┐        │   CANS   │

│  └──────────────┘  └──────────────┘        │  │ ▓▓ │  │
│  ┌──────────────┐  ┌──────────────┐        │  │ ▓▓ │  │
│  │ ManCave  ⦿──│  │ Foyer    ⦿──│        │  │ ░░ │  │
│  └──────────────┘  └──────────────┘        │  │ ░░ │  │

└────────────────────────────────────────────┴──────────┘

- Left ~75%: 2×3 grid of large tiles (each roughly a third of the screen wide).
- Right ~25%: one tall vertical slider ("Cans by Row").
- Aesthetic: dark charcoal background, rounded Mushroom tiles (card-mod), warm amber
  icon/slider glow when a zone is on, large type. No header, no clock, no badges.

### Card stack (all already installed via HACS)

- **lovelace-mushroom** — zone tiles (mushroom-light-card, mushroom-entity-card)
- **my-cards (my-slider)** — vertical row slider
- **lovelace-layout-card** — grid
- **lovelace-card-mod** — styling

## Zone tiles

Mushroom light cards with `show_brightness_control: true`: tap anywhere toggles; the tile
body is a drag-to-dim slider. Bathroom is a switch, so it gets a Mushroom entity card
styled identically (toggle only).

| Tile | Entity | Dimming |
|---|---|---|
| Basement Cans | `light.basement_cans` | yes |
| Bar Lights | `light.bar_lights` | yes |
| ManCave | `light.mancave` | yes |
| Foyer | `light.basement_foyer` (**new** light group = `light.bathroom_foyer_a` + `light.bathroom_foyer_b`) | yes |
| Bathroom | `switch.basement_bathroom_lights` | toggle only |

The foyer's ISY load switch (`light.basement_br_foyer_sw`) is being replaced; the two
Hue foyer lights are currently `unavailable` and will come online after the swap. The
tile renders greyed until then — no dashboard change needed afterward.

## Cans-by-row slider

A vertical my-slider bound to a new helper:

- **`input_number.basement_cans_rows`** — min 0, max 4, step 1.

Semantics: value = number of rows on, counting contiguously from the **rear**. Sliding
down turns rows off **front to back** (rear of the room stays lit longest):

| Value | Rows on |
|---|---|
| 4 | 1, 2, 3, 4 (all) |
| 3 | 2, 3, 4 |
| 2 | 3, 4 |
| 1 | 4 only |
| 0 | none |

### Row → entity map (operator-confirmed)

| Row (front→back) | Entities |
|---|---|
| 1 (front) | `light.0xc890a81f69ed0000` (Left Front Row), `light.0xc890a81f577b0000` (Center Front Row), `light.right_row_1_front` |
| 2 | `light.0xc890a81eb5df0000` (Left Row 2), `light.0xc890a81f6ff30000` (Center Middle), `light.right_row_2_3` |
| 3 | `light.left_row_3_3`, `light.right_row_3_3` (no center can) |
| 4 (rear) | `light.left_rear_row_3` (Left Rear Row), `light.0xc890a81f5a530000` (Center Rear Row), `light.0xc890a81ed77e0000` (Right Row 4 Rear) |

### Automations

1. **Forward** (`basement_cans_rows_apply`): on `input_number.basement_cans_rows` state
   change, `homeassistant.turn_on`/`turn_off` each row per the table. Calls are
   idempotent; already-correct rows are harmless no-ops.
2. **Reverse** (`basement_cans_rows_sync`): on any row light state change, compute the
   largest N such that rows (5−N)…4 are all on, and `input_number.set_value` to N.
   Mixed/non-conforming states round down. Setting an unchanged value produces no state
   change, so the forward/reverse pair cannot loop.

The reverse automation keeps the slider truthful when cans change via the Cans tile,
scenes, Hue/Z2M, or anything else.

## All Off

One button-style tile: `homeassistant.turn_off` targeting the five zone entities plus
all 11 individual row lights (idempotent; covers rows even if the cans group doesn't).
The reverse automation snaps the slider to 0. No confirmation step — turning lights off
is harmless.

## Repo layout & tooling

New `ha/basement_tablet/` (standalone from PAE — PAE's HA client is read-only enforced),
mirroring `ha/weather/`:

- `helpers.py` — creates `input_number.basement_cans_rows`, the `light.basement_foyer`
  group, and the two automations (REST `POST /api/config/automation/config/<id>`).
- `build_dashboard.py` — generates `dashboard.json` from the entity/row tables above.
- `deploy_dashboard.py` — creates the `basement-tablet` dashboard if missing and pushes
  the config via WS `lovelace/config/save` (`max_msg_size=0`).
- `README.md` — run order and commands.

All scripts support `--dry-run` and print what they would write. **No write to HA
executes without explicit operator confirmation at deploy time** (operator hard rule).

## Error handling

- Unavailable entities (foyer until the switch swap, any dropped Zigbee can) render
  greyed on Mushroom cards; taps on them are no-ops.
- Automations acting on unavailable rows: service calls fail soft (no crash, other rows
  still actuate).
- The dashboard is generated: to change it, edit `build_dashboard.py`, rebuild, redeploy.

## Testing / verification

- `--dry-run` output reviewed before every deploy.
- Row-mapping logic in `build_dashboard.py`/`helpers.py` covered by unit tests
  (pure functions: value → row on/off sets, row states → value).
- Manual end-to-end on the tablet: each tile toggles and dims its zone; slider walks
  rows front→back both directions; external change (Hue app / Cans tile) updates the
  slider; All Off zeroes everything.
